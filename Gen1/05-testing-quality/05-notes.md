# Модуль 5. Тестирование и обеспечение качества кода

## Цель модуля
Показать, как ИИ ускоряет подготовку тестов, но качество определяется инженерным дизайном тест-кейсов.

## Длительность
4 часа (50 мин теория, 70 мин демонстрации, 120 мин практика)

## Темы модуля
- Генерация unit и integration тестов.
- Улучшение покрытия и анализ “пустого” покрытия.
- Property-based тестирование.
- Генерация edge-case сценариев.
- Тесты для C++/C# и языко-специфические нюансы.

## Демонстрация
1. Сгенерировать unit-тесты для функции с явными ветвлениями.
2. Показать типичную проблему: тест есть, а баг не ловит.
3. Добавить edge-cases через ИИ и повысить полезность тестов.
4. Сгенерировать integration тест против тестового API.
5. Сравнить реальное качество до/после по defect detection.

## Практика 1
**Задача**: участник берет модуль из MVP и создает набор unit + integration тестов.

**Критерии приемки**:
- Тесты покрывают happy path и минимум 3 edge-case.
- Ошибочные входные данные проверяются явно.
- Тесты детерминированы и не флапают локально.

## Практика 2 (advanced)
**Задача**: применить property-based подход к функции с ограничениями.

**Критерии приемки**:
- Описаны свойства, которые должны всегда выполняться.
- Найден минимум 1 неожиданный контрпример.
- Исправление подтверждено тестом.

## Senior/EM акценты
- Метрика “coverage %” не равна качеству.
- Нужны тест-пирамиды и стратегия стоимости поддержки.
- Полезно внедрять шаблоны AI-assisted тестирования в CI.

## Термины для объяснения
- **Unit-тест**: проверяет одну небольшую часть логики.
- **Integration-тест**: проверяет работу нескольких компонентов вместе.
- **Edge-case**: редкий, но важный пограничный сценарий.
- **Flaky test**: нестабильный тест, который “иногда падает”.

## Типичные ошибки
- Слепо принимать тесты модели без чтения assertions.
- Не отделять бизнес-логику от инфраструктуры, усложняя тесты.
- Не контролировать фикстуры и состояние окружения.

## Quality gates: линтеры, форматтеры, SAST, AI-review

> Подробно — `05-chapter.md` §5.11a. Здесь — что должно прозвучать на занятии.

### Зачем это в Module 5
Тесты ловят дефекты поведения; линтеры/SAST/SCA ловят дефекты кода **до** запуска. В AI-эпоху без статических гейтов команда тонет в шуме: модель за минуту генерирует столько кода, сколько человек ревьюит за день.

### Эмпирика 2024–2026 (распределение AI-проблем)
- 30–50% — синтаксис/импорты/типы → формат + линтер + typecheck.
- 15–25% — security smells (SQL-injection через f-strings, hardcoded secrets) → SAST + secret-scan.
- 5–15% — slopsquatting, CVE в зависимостях → SCA.
- 20–30% — семантика/архитектура → тесты и человеческое ревью.

### Пять обязательных гейтов
1. **Форматтер** — Black/Ruff format, Prettier, dotnet format, clang-format, gofmt, rustfmt.
2. **Линтер** — Ruff, ESLint, Roslyn analyzers, golangci-lint, clippy, clang-tidy.
3. **Type checker** — mypy --strict, tsc --strict, компилятор с -Werror.
4. **SAST** — Bandit, Semgrep, CodeQL, SonarQube, Snyk Code.
5. **SCA + secret-scan** — pip-audit, npm audit, Trivy, gitleaks, detect-secrets.

### Где запускать
- **pre-commit (локально)**: формат + линтер --fix + secret-scan. Время ≤ 5 сек, иначе обходят.
- **CI на PR**: формат-чек + линтер + typecheck + SAST + SCA + тесты. Время ≤ 5 мин до фидбека.
- **CI nightly**: полный SAST (CodeQL polyglot), SCA + license-check, mutation testing.

### Линтер как verifier-loop для агента
- `post-tool-call` hook (см. модуль 3) запускает `ruff format` + `ruff check --fix` после каждой правки.
- Агент в 70–90% случаев исправляет lint-ошибки без человека.
- Без этого hook'а L4–L5 агенты деградируют в угадайку.

### AI-review как дополнительный гейт _(as of Q2 2026)_
- Инструменты: CodeRabbit, GitHub Copilot Code Review, Greptile, PR-Agent.
- Полезно: стилистика, AI-типичные уязвимости, summary PR, предложение тестов.
- Не заменяет: архитектурное ревью, бизнес-логика, финальный approve owner'а.

### Демонстрация (15–20 мин)
1. Сгенерировать функцию с f-string SQL — Bandit/Semgrep ловит.
2. Сгенерировать импорт несуществующего пакета (`import requests_async`) — pip-audit/линтер ловит.
3. Сгенерировать функцию с `secret_key="changeme"` — gitleaks/detect-secrets ловит.
4. Включить hook на `post-tool-call` и показать авто-исправление линтера агентом.
5. Запустить CodeRabbit на тестовом PR и обсудить, что бот нашёл и что пропустил.

### Типичные ошибки участников
- «Линтер потом» — через год не настроен, AI-долг утроен.
- pre-commit > 5 сек → обходят через `--no-verify`.
- AI-review с auto-approve → 30% мерджей без человека.
- Запуск SAST только nightly → проблемы доходят до main.
- Доверие AI-review без code-owner'а → размывается accountability.

### Senior / EM-фокус
- Стоимость владения базового гейт-сета: 2–4 рабочих дня инженера на старте + ~4 недели на typecheck strict-режим в legacy.
- Метрика «процент PR'ов, заблокированных автоматическими гейтами» — индикатор зрелости pipeline'а.
- AI-review = дополнительный, не заместительный ревью.

## Артефакт модуля
Тестовый пакет для одного сервиса + отчет "что именно эти тесты защищают".
