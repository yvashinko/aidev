# Модуль 2. Prompt Engineering для разработчиков

## Цель модуля
Научить участников формулировать запросы так, чтобы получать инженерно полезный, проверяемый и повторяемый результат.

## Длительность
4 часа (60 мин теория, 60 мин демонстрации, 120 мин практика)

## Ключевые элементы эффективного промпта
- Роль модели (кто ты в этой задаче).
- Контекст (код, ограничения, стек, бизнес-цель).
- Задача (что именно нужно сделать).
- Формат ответа (структура, язык, ограничения).
- Критерии качества (что считается хорошим результатом).

## Что показать преподавателю
- Роль `system prompt`: задает поведение и границы.
- Few-shot: как примеры вход/выход повышают стабильность.
- Chain-of-thought: когда полезно просить пошаговый разбор.
- Контекст-менеджмент: как дозировать данные и избежать “шума”.

## Живая демонстрация
1. Некачественный промпт “напиши API” и его слабый результат.
2. Рефакторинг промпта по шаблону и заметное улучшение выхода.
3. Few-shot для генерации SQL с заданным стилем и требованиями к индексации.
4. Промпт на объяснение сложного алгоритма под junior-аудиторию.

## Практика 1
**Задача**: улучшить 3 “плохих” промпта (код, SQL, документация) до production-уровня.

**Критерии приемки**:
- Каждый промпт содержит явные критерии качества.
- Ответ модели проходит минимальную ручную проверку.
- Участник объясняет, почему новый промпт лучше старого.

## Практика 2 (senior-фокус)
**Задача**: спроектировать командный шаблон промптов для code review/bugfix/doc.

**Критерии приемки**:
- Есть версионируемый шаблон в репозитории.
- Есть правило, когда шаблон использовать нельзя.
- Есть метрика эффективности (например, время на PR review).

## Практические шаблоны, которые стоит дать
- Генерация функции с ограничениями по сложности и обработке ошибок.
- Объяснение алгоритма “в двух режимах”: для junior и для senior.
- Генерация SQL с обязательной проверкой плана запроса.
- Разбор legacy-кода с предложением безопасного рефакторинга.

## Типичные ошибки
- Слишком общий запрос без ограничений.
- Нет формата ответа (модель отвечает “эссе”, а нужен diff/список).
- Нет условия “если не хватает данных, задай уточняющие вопросы”.

## Сессии работы: применение и автоматизация

> Подробно — `02-chapter.md` §2.5a. Здесь — то, что нужно проговорить на занятии.

### Ключевые понятия простыми словами
- **Сессия (chat session)** — диалог с моделью с общей историей и состоянием. Промпт = спецификация задачи, сессия = окружение исполнения.
- **Hand-off** — структурированный summary текущей сессии, который скармливается в первое сообщение следующей сессии вместо полной истории.
- **Rules-файл (`AGENTS.md`, `.cursorrules`, `CLAUDE.md`)** — текстовый файл в корне репозитория, авто-добавляемый в контекст каждой сессии этого проекта.
- **Skill** — модульная единица знания + поведения, активируемая агентом по запросу.
- **Hook** — скрипт на события сессии (`pre-prompt`, `post-tool-call`, `pre-commit`).

### Когда сессию пора закрывать
- модель забывает правила, которые установили в начале;
- история длиннее, чем содержательная задача;
- сменилась тема — а контекст предыдущей всё ещё «давит»;
- prompt cache hit-rate начал падать (видно в счёте API).

### Демонстрация (10–15 мин)
1. Показать «бесконечную сессию» на 30+ turns — деградацию ответов и счёта.
2. Сделать hand-off в новую сессию через шаблон summary.
3. Сравнить качество ответа на одинаковой задаче «новая чистая сессия» vs «старая загрязнённая».
4. Показать `AGENTS.md` в корне репо и как он авто-инжектируется.
5. Показать сниппет `/handoff` и hook на экспорт транскрипта.

### Шаблон hand-off
```text
Контекст: [3-5 строк, что делаем и зачем]
Сделано: [bullet list решений и артефактов]
Открытые вопросы: [bullet list]
Следующий шаг: [одна задача]
Правила, которые нужно помнить: [см. AGENTS.md + session-specific]
```

### Способы сохранения сессий _(as of Q2 2026)_
- **ChatGPT / Claude.ai**: встроенная история, export по запросу.
- **Cursor Chat**: `Export Chat`, расширение **SpecStory** для авто-сохранения в `.specstory/history/`.
- **Claude Code**: авто-сохранение в `.claude/history/`, команда `/save`.
- **Aider**: auto-commit чата в `.aider.chat.history.md`.
- **API-интеграции**: LLMOps-стек (Langfuse, Helicone, LangSmith).

### Пять уровней автоматизации
1. **Rules-файлы** (`AGENTS.md`) — базовый уровень.
2. **Prompt-сниппеты** в `.cursor/rules/` или `.github/prompts/`.
3. **Skills** — модульные правила поведения агента.
4. **Hooks** — скрипты на события сессии.
5. **Backend-middleware** (LiteLLM-proxy, Helicone) — инжекция rules, кеширование, transcripts в LLMOps.

### Типичные ошибки участников
- Одна «бесконечная» сессия на весь рабочий день.
- Hand-off через «прочитай мою предыдущую переписку» вместо structured summary.
- Транскрипты commited в репо без secret-scan → утечка токенов и PII.
- Rules-файл на 800+ строк, который никто не читает (kitchen-sink).
- Шаблоны промптов в личном Notion, а не в репозитории.

### Практика 3 (опционально, командная)
**Задача**: за 30 минут команда оформляет минимальный стартовый набор для своего учебного репозитория: `AGENTS.md`, два сниппета промпта, шаблон `/handoff`, pre-commit hook на secret-scan транскриптов.

**Критерии приемки**:
- `AGENTS.md` ≤ 200 строк, содержит стек, версии, naming, error style.
- Два сниппета покрывают «генерация функции» и «генерация теста».
- Шаблон `/handoff` фиксирован в репозитории.
- Pre-commit hook поднимает ошибку на тестовом транскрипте с фейковым `OPENAI_API_KEY=sk-...`.

## Артефакт модуля
Набор `Prompt Playbooks` для 5 инженерных сценариев: feature, bugfix, review, test, doc.
