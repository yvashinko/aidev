# Модуль 4. Debugging и анализ логов

## Цель модуля
Научить участников использовать ИИ как “ускоритель анализа”, но не как замену инженерной диагностике.

## Длительность
4 часа (45 мин теория, 75 мин демонстрации, 120 мин практика)

## Ключевые темы
- Анализ stack trace.
- Анализ логов приложения и инфраструктуры.
- Поиск root cause.
- Генерация fixes и проверка регрессий.
- Анализ состояния системы и production-ошибок.

## Сценарий демонстрации
1. Дать участникам “грязный” лог с несколькими симптомами.
2. Попросить модель выделить вероятные причины.
3. Сравнить гипотезы модели с реальными метриками/логами.
4. Сгенерировать патч и обсудить риски.
5. Подтвердить fix через тест или повтор сценария.

## Практика 1
**Задача**: разобрать инцидент “API отвечает 500 на часть запросов”.

**Критерии приемки**:
- Составлена timeline событий инцидента.
- Выделены symptom vs root cause.
- Предложен и проверен минимум один fix.

## Практика 2 (production-minded)
**Задача**: подготовить postmortem с участием ИИ и ручной валидации.

**Критерии приемки**:
- Есть раздел “что модель предложила и почему часть отклонена”.
- Есть action items по предотвращению повторения.
- Есть owner и срок на каждый action item.

## На что делать акцент
- Модель помогает формировать гипотезы, но не доказывает их.
- Корректный debug начинается с воспроизводимости.
- Логи без структуры и correlation id сильно ограничивают полезность ИИ.

## Простые термины для junior-вопросов
- **Stack trace**: цепочка вызовов, где произошла ошибка.
- **Root cause**: первопричина, а не внешнее проявление.
- **Incident**: событие, влияющее на стабильность сервиса.
- **Postmortem**: разбор инцидента после восстановления.

## Типичные ошибки
- Лечат сообщение об ошибке, а не причину.
- Генерируют fix без сценария воспроизведения.
- Не проверяют влияние исправления на соседние компоненты.

## Артефакт модуля
Incident report: timeline, root cause, fix, тест проверки, preventive actions.
