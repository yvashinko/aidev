# Модуль 6. Документация и архитектурные решения

## Цель модуля
Научить участников использовать ИИ для ускорения документации, сохраняя инженерную точность и прозрачность решений.

## Длительность
3.5 часа (45 мин теория, 60 мин демонстрации, 105 мин практика)

## Темы
- Генерация README.
- Генерация API-документации.
- Генерация ADR.
- Архитектурные обсуждения с ИИ.

## Демонстрация
1. Из существующего кода сгенерировать черновик README.
2. Улучшить структуру: quick start, env vars, troubleshooting.
3. Сформировать ADR по выбору БД/кеша.
4. Проверить документ на полноту и “проверяемость”.

## Практика 1
**Задача**: подготовить README для MVP-сервиса и сделать его пригодным для onboarding.

**Критерии приемки**:
- Новый разработчик поднимает проект по README за 15 минут.
- Описаны зависимости, запуск и базовые сценарии проверки.
- Есть раздел “известные ограничения”.

## Практика 2
**Задача**: оформить ADR по одному спорному техническому решению.

**Критерии приемки**:
- Зафиксированы контекст, варианты, выбранное решение и последствия.
- Есть причины отклонения альтернатив.
- Есть критерий пересмотра ADR в будущем.

## Что проговорить как Engineering Manager
- Хорошая документация снижает bus factor и ускоряет delivery.
- ADR помогает не повторять одни и те же архитектурные споры.
- Документация — часть Definition of Done, а не “дополнительная активность”.

## Термины для junior
- **README**: стартовая инструкция по проекту.
- **ADR**: запись архитектурного решения с аргументами.
- **Trade-off**: осознанный компромисс между несколькими целями.

## Типичные ошибки
- Документация описывает “как должно быть”, а не “как сейчас есть”.
- Нет конкретных команд запуска и проверки.
- ADR без последствий и критериев пересмотра.

## Артефакт модуля
Комплект `README + API doc + ADR`, пригодный для внутреннего engineering wiki.
