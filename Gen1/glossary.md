# Общий глоссарий

## Модуль 1 — LLM fundamentals

**LLM (Large Language Model)** — параметрическая функция, аппроксимирующая распределение следующего токена по контексту.

**Токен** — фрагмент текста из фиксированного словаря, которым оперирует модель.

**Контекстное окно** — жёсткий лимит суммы input + output токенов одного запроса.

**Knowledge cutoff** — момент времени, после которого данные не попали в pretraining-корпус.

**Галлюцинация** — уверенный, но фактически неверный ответ модели.

**Temperature** — параметр сэмплинга; T=0 → детерминизм.

**KV-cache** — сохранённые K и V предыдущих токенов для ускорения decode.

**C-C-S-I-M** — пять осей оценки ответа: Correctness, Completeness, Safety, Idiomaticity, Maintainability.

## Модуль 2 — Prompt Engineering

**R-C-T-F-Q** — Role, Context, Task, Format, Quality criteria: пять элементов промпта.

**System prompt** — сообщение, задающее стабильное поведение модели.

**Few-shot** — 2–5 примеров «вход → выход» в промпте.

**Chain-of-Thought (CoT)** — пошаговое рассуждение перед ответом.

**Prompt injection** — инструкция во входных данных, переопределяющая system prompt.

**Structured outputs** — гарантированное соответствие выхода заданной схеме.

## Модуль 3 — Генерация кода

**Specification (for AI-MVP)** — компактный input-контракт на 1–3 страницы.

**AGENTS.md** — файл правил для AI-ассистентов в репозитории.

**Plan-Implement-Verify (PIV)** — цикл инкрементальной AI-сборки.

**Prompt-driven architecture drift** — постепенное смещение архитектуры под влиянием умолчаний модели.

**AI provenance** — документирование модели, промптов и доли ручных изменений в PR.

## Модуль 4 — Debugging

**HDD (Hypothesis-Driven Debugging)** — диагностика, где каждое действие проверяет явную гипотезу.

**MRE (Minimal Reproducible Example)** — компактная воспроизводимая версия бага.

**MTTR** — среднее время восстановления после инцидента.

**Postmortem** — структурированный разбор инцидента после восстановления.

**Blameless culture** — фокус на системе, допустившей ошибку, а не на человеке.

## Модуль 5 — Тестирование

**Observable behavior** — внешний результат работы кода, доступный клиенту.

**Mutation testing** — техника внесения малых изменений в код для оценки качества тестов.

**Mutation score** — доля мутантов, убитых тестами.

**Property-based testing** — тестирование инвариантов над классами входов.

**Flaky test** — нестабильный тест, исход которого зависит от не-входных факторов.

## Модуль 6 — Документация

**ADR (Architecture Decision Record)** — immutable-документ, фиксирующий одно архитектурное решение.

**MADR** — Markdown Any Decision Records, лёгкий формат ADR.

**Doc-as-code** — документация в репозитории, редактируется через PR, собирается через CI.

**TTFS (Time-to-first-success)** — время от старта нового инженера до первого успешного smoke-результата.

**Drift detector** — проверка расхождения между документом и кодом.

## Модуль 7 — Локальные модели

**Open-weights model** — модель с публично доступными весами.

**Quantization** — снижение разрядности весов (fp16 → int8 / int4).

**GGUF** — формат весов для llama.cpp / Ollama / LM Studio.

**VRAM** — видеопамять GPU.

**TCO (Total Cost of Ownership)** — суммарная стоимость владения за период.

## Модуль 8 — RAG

**RAG (Retrieval-Augmented Generation)** — генерация с предварительным поиском по внешней базе.

**Embedding** — числовое представление текста для поиска похожего контента.

**Chunking** — разбиение документа на фрагменты для индексации.

**Hybrid search** — комбинация dense (vector) и sparse (BM25) retrieval.

**Reranking** — второй проход top-K через более точную модель.

**Citation grounding** — цитирование источников в ответе модели.

**Faithfulness** — доля ответов, опирающихся на retrieved context.

**MCP (Model Context Protocol)** — протокол подключения LLM-приложений к источникам данных и инструментам.
