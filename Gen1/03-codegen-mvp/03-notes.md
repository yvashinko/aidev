# Модуль 3. Генерация кода и MVP проектов

## Цель модуля
Показать, как использовать ИИ для ускоренного создания сервисов, API и CLI без деградации качества архитектуры.

## Длительность
5 часов (60 мин теория, 90 мин демонстрации, 150 мин практика)

## Основные темы
- Генерация сервисов и API.
- Генерация CLI-инструментов.
- Генерация дизайна и структуры MVP.
- Инкрементальная сборка backend MVP.

## Демонстрация (сквозной сценарий)
1. Постановка задачи: “MVP сервиса задач с REST API”.
2. Генерация каркаса проекта и модулей.
3. Генерация endpoint-ов и валидации входных данных.
4. Генерация CLI для админ-операций.
5. Ручная инженерная доработка: ошибки, логирование, конфиг.
6. Финальная проверка: smoke-тесты + линтер + запуск.

## Что важно проговорить
- ИИ ускоряет старт, но не отменяет архитектурную ответственность.
- Любой сгенерированный код проходит те же quality gates, что и ручной.
- В MVP важно “минимально работающее”, а не “максимально красивое”.

## Практика 1 (командная)
**Задача**: за 90 минут собрать backend MVP из 3 endpoint-ов + базовой персистентности.

**Критерии приемки**:
- API поднимается и отвечает корректными HTTP-кодами.
- Есть базовая обработка ошибок и валидация.
- Есть README со сценариями запуска.

## Практика 2 (индивидуальная)
**Задача**: сгенерировать CLI-инструмент для импорта/экспорта данных в MVP.

**Критерии приемки**:
- CLI имеет минимум 2 команды и `--help`.
- Ошибки пользователя обрабатываются явно.
- Есть примеры использования в документации.

## Senior-фокус
- Где провести границы между generated и handwritten code.
- Как не допустить “prompt-driven architecture drift”.
- Как встроить ИИ-генерацию в workflow PR и review.

## Типичные ошибки
- Генерируют слишком много кода за один шаг, теряя управляемость.
- Не фиксируют архитектурные решения по мере генерации.
- Игнорируют требования по безопасности (секреты, валидация, auth).

## IDE-агенты, skills, rules: таксономия для занятия

> Подробно — `03-chapter.md` §3.8 (rules) и §3.8a (агенты, skills, hooks).

### Пять уровней автономности AI-ассистента _(as of Q2 2026)_
| Уровень | Что делает | Примеры |
|---|---|---|
| L1 — Tab / автокомплит | Достраивает строку | Cursor Tab, Copilot, JetBrains AI |
| L2 — Chat | Отвечает в боковой панели | Cursor Chat, Copilot Chat, Cody |
| L3 — Inline edit | Правит выделенный фрагмент | Cursor Cmd-K, Copilot Edit |
| L4 — Agent | Выполняет 3–30 шагов сам | Cursor Agent, Claude Code, Aider, Codex CLI |
| L5 — Background agent | Асинхронный агент в облаке | Devin, Cursor Background, Codex (cloud), Copilot Workspace |

Главный закон: **уровень полезен ровно настолько, насколько быстр верификатор** (тесты, линтер, типы).

### Три типа командных артефактов
- **Rules** (`AGENTS.md`, `.cursorrules`) — что модель **не должна** делать. Декларативные, инжектируются всегда. ≤ 200 строк.
- **Skills** (`.cursor/skills/`, Claude Skills) — **как** модель должна делать класс задач. Активируются по триггеру.
- **Hooks** (`hooks.json`) — что должно произойти **вокруг** работы модели (формат после правки, secret-scan перед коммитом, экспорт транскрипта).

### Демонстрация (15–20 мин)
1. Показать `AGENTS.md` в учебном репозитории и поведение агента **с** ним и **без**.
2. Заведение одного skill'а под класс задач "добавить endpoint" и его срабатывание.
3. Hook на `post-tool-call` с авто-форматированием и hook на `pre-commit` с secret-scan.
4. Пример L5-сценария (если есть доступ): фоновый агент закрывает Issue → PR.

### Что объяснить на занятии
- Rules — foundation, без них skills и hooks теряют смысл.
- Skill заводится **только** после того, как класс задач встретился ≥ 3 раз и каждый раз требовал > 5 минут промптинга.
- Hook исполняет код. Враждебный hook в PR — это RCE. Hook ≠ просто конфиг, hook = production-код.
- Граница «человек vs агент»: человек владеет контрактами, агент — реализацией.

### Типичные ошибки участников
- 800-строчный `AGENTS.md` («kitchen-sink»).
- 50 skill'ов за первый месяц «потому что красиво».
- Hook'и без security review.
- Использование L4–L5 без rapid verifier (тестов нет → агент уходит в drift).
- Запрет AI вместо governance: команды находят теневое использование.

### Практика 3 (опционально, индивидуальная)
**Задача**: на учебном MVP завести `AGENTS.md` ≤ 200 строк, один skill, один pre-commit hook.

**Критерии приемки**:
- `AGENTS.md` фиксирует стек, версии, naming, error style, security baseline.
- Skill описан в формате `name / description / trigger / steps / constraints` и срабатывает хотя бы на одном тестовом промпте.
- Hook поднимает ошибку на тестовом коммите с фейковым `OPENAI_API_KEY=sk-...`.

## Артефакт модуля
Рабочий MVP (API + CLI + README) с понятным списком допущений и технических долгов.
