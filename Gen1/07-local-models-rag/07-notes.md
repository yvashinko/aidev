# Модуль 7. Локальные модели для разработки и RAG

## Цель модуля
Научить участников запускать локальные LLM и строить практичные сценарии поиска знаний по документации и кодовой базе.

## Длительность
5 часов (60 мин теория, 90 мин демонстрации, 150 мин практика)

## Темы
- Обзор локальных LLM и критерии выбора.
- Работа с Ollama.
- Работа с LM Studio.
- Локальный code review.
- RAG по документации и репозиторию.
- Подключение внутренней базы знаний.

## Демонстрация 1: локальный запуск
1. Установка модели через Ollama/LM Studio.
2. Проверка latency, качества и требований к железу.
3. Сравнение с облачной моделью на одинаковом промпте.

## Демонстрация 2: RAG
1. Подготовка корпуса документов/кода.
2. Индексация и retrieval.
3. Ответ модели с цитированием источников.
4. Проверка, что ответ реально опирается на базу знаний.

## Практика 1
**Задача**: локально развернуть модель и выполнить code review одного модуля.

**Критерии приемки**:
- Review содержит минимум 5 замечаний с приоритетами.
- Минимум 2 замечания подтверждены вручную.
- Отмечены ложноположительные срабатывания.

## Практика 2
**Задача**: собрать мини-RAG для внутренней документации проекта.

**Критерии приемки**:
- Система возвращает релевантные источники.
- Ответы содержат ссылки на конкретные документы/файлы.
- Участник демонстрирует сценарий “не знаю ответ, но могу быстро найти”.

## Архитектурные акценты для senior
- Data governance: какие документы можно индексировать.
- Security: защита секретов и приватных данных в локальном контуре.
- Стоимость владения: железо, поддержка, обновление моделей.
- Когда локальный стек лучше облака, и наоборот.

## Простые термины
- **RAG**: подход, где модель отвечает с опорой на внешние источники.
- **Embedding**: числовое представление текста для поиска похожего контента.
- **Retrieval**: выбор релевантных фрагментов перед генерацией ответа.

## Типичные ошибки
- Индексация “всего подряд” без политики доступа.
- Отсутствие оценки качества retrieval.
- Ожидание, что локальная модель всегда дешевле и точнее.

## Квантование (quantization) — конспект для занятия

> Подробно — `07-chapter.md` §7.4. Здесь — то, что обязательно прозвучит на занятии.

### Простыми словами
- **Квантование** — снижение разрядности весов модели: fp16 → int8 / int4 / int3 / int2.
- Цель: размер модели падает в 2–8×, скорость на CPU/GPU растёт, качество падает нелинейно.
- Аналогия: фотография в RAW (16 бит/канал) → JPEG (8 бит/канал) — файл легче, на глаз почти не видно, но на мелких деталях разница есть.

### Ключевые форматы _(as of Q2 2026)_
- **GGUF q4_K_M** — золотой стандарт для llama.cpp / Ollama / LM Studio. ~0.55 ГБ на 1B параметров, потеря качества vs fp16 — 2–6%.
- **GGUF q5_K_M** — для тех, кто готов отдать +25% памяти за +2% качества.
- **AWQ int4** — лучше для GPU-инференса (vLLM) на 32B+ моделях.
- **GPTQ int4** — конкурент AWQ, обычно чуть слабее по качеству.
- **q3 и ниже** — рискованно для code-задач: reasoning заметно деградирует.

### Эвристика VRAM
- fp16: ~2 ГБ VRAM на 1B параметров.
- q4: в 4× меньше, ~0.5 ГБ на 1B.
- q5: ~0.6 ГБ на 1B.
- + 1–10 ГБ на KV-cache (зависит от контекста и batch).

### Что объяснить участникам
- «70B на 24 ГБ» через q2 — это академический эксперимент, не workhorse.
- Стандартный workhorse 2026: Qwen2.5-Coder-32B q4_K_M на 32–48 ГБ GPU с 16k контекстом.
- Apple Silicon M3/M4 Pro/Max/Ultra с unified memory 36–192 ГБ — единственное массовое не-NVIDIA решение.

## Hugging Face Hub — конспект для занятия

> Подробно — `07-chapter.md` §7.3a.

### Что это
**HF Hub** — крупнейший публичный hub: 1.5+ млн моделей, 350+ тыс. датасетов, 250+ тыс. Spaces _(as of Q2 2026)_. Три роли в курсе:
1. Источник весов (официальные релизы Meta, Mistral, Qwen, DeepSeek, Google).
2. Источник GGUF-квантизаций (community: `bartowski/...`, `QuantFactory/...`).
3. Источник датасетов и benchmark'ов (HumanEval, MBPP, SWE-bench).

### Обязательные команды (показать на занятии)
```bash
huggingface-cli login
huggingface-cli download Qwen/Qwen2.5-Coder-7B-Instruct --local-dir ./qwen
ollama run hf.co/bartowski/Qwen2.5-Coder-32B-Instruct-GGUF:Q4_K_M
```

### Лицензии — не пропускать
- **Apache 2.0 / MIT** — без ограничений (Mistral, Qwen, Phi, частично Gemma).
- **Llama Community License** — commercial с ограничениями (700M MAU lock, anti-distillation).
- **Gemma Terms** — commercial с restrictions.
- **Research-only / gated** — для commercial use требует legal review.

### Типичные ошибки
- «Open-weights = можно всё» — нет: читать лицензию обязательно.
- Слепое следование HF Open LLM Leaderboard без custom eval (см. §7.10).
- Скачивание полных safetensors там, где хватит GGUF.

## MCP (Model Context Protocol) — конспект для занятия

> Подробно — `07-chapter.md` §7.12.

### Что это и зачем
- **MCP** — открытый протокол (Anthropic, ноябрь 2024) для подключения LLM-приложений к источникам данных и инструментам.
- Аналогия: «USB-C для AI» / «LSP, но для AI-контекста».
- До MCP: каждый IDE-агент имел свою интеграцию с GitHub, Postgres, Jira → N×M интеграций.
- С MCP: каждый клиент реализует протокол один раз, каждый источник — сервер один раз → N+M.

### Архитектура (на занятии — диаграмма)
```text
MCP-клиент (Cursor, Claude Desktop, Codex, Cline)
    ↕ JSON-RPC over stdio / HTTP+SSE
MCP-сервер (github, postgres, file-system, slack, jira, custom)
    ↕
Реальная система (API, БД, FS)
```

### Три типа возможностей сервера
- **Resources** — read-only данные (содержимое файла, запись в БД, страница вики).
- **Tools** — действия с побочным эффектом (создать issue, выполнить SQL, записать файл).
- **Prompts** — готовые workflow-шаблоны от сервера.

### Демонстрация (15–20 мин)
1. Показать `~/.cursor/mcp.json` с подключённым `github` MCP-сервером.
2. В чате попросить «покажи open issues в репо X» — модель видит resource из MCP.
3. Запустить простейший custom MCP-сервер на Python (FastMCP) с одним tool.
4. Показать каталог modelcontextprotocol.io и Smithery.

### Безопасность — обязательный блок
- MCP-сервер исполняет код. Враждебный сервер = RCE на машине разработчика.
- **Whitelist** серверов (только из доверенных registry или прошедших code review).
- **Sandbox** через Docker.
- **Indirect prompt injection** через resource-content (см. §7.11).
- Аудит-лог tool calls в корпоративном контуре.

### Что приходит «дополнением» к MCP _(as of Q2 2026)_
- **A2A (Agent-to-Agent)** от Google (апрель 2025) — общение агентов между собой; **дополняет** MCP, не заменяет.
- **OpenAI Apps SDK** — встраивание приложений в ChatGPT; внутри использует MCP.
- **Function calling** — оригинальный механизм tool use в одном API-вызове; не конкурирует с MCP.

«Убийцы MCP» в 2026 не ожидается: он занял ту же роль, что HTTP в web и LSP в IDE.

### MCP vs RAG
- **RAG** достаёт **знание** из больших корпусов (документация, код).
- **MCP** даёт доступ к **операционным системам** (GitHub, БД, мониторинг).
- Зрелый AI-ассистент 2026 использует оба.

### Типичные ошибки участников
- Подключение случайных MCP-серверов из marketplace без code review.
- Tool с правом `DROP TABLE` («ну для тестов»).
- Запуск серверов без sandbox.
- Игнорирование indirect prompt injection через issue/wiki.

## Артефакт модуля
Локальный AI-ассистент для проектной документации с демонстрацией поиска по коду.
